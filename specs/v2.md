# Parallax CLI — V2 Specification

**Version:** 2.0  
**Date:** 2026-04-15  
**Status:** North Star — authoritative reference for the V2 build

This document defines the command surface, data contracts, capability modes, test requirements, and onboarding behavior for Parallax CLI V2. It supersedes V1's ad-hoc command set. Any deviation from this spec requires an explicit decision entry in DEV_LOG.md.

---

## Background and V1 Problems

V1 shipped a flat bag of verbs with no organizing principle. `parallax run` became a catch-all that absorbed too much logic. `parallax animate` was a misleading name for what is actually Ken Burns motion on stills. There was no clean separation between generation commands, editing commands, and pipeline orchestration. Commands lacked consistent contracts for what they read, what they write, and what exit codes mean.

V2 fixes this with a three-tier namespace: **generation** (`generate`), **editing** (`edit`, `trim`, `manifest`), and **pipeline** (`ingest`, `compose`, `script`). Every command has a declared contract. Every command supports `--json` for agentic use.

## Command Contracts

Each command entry follows this format: what it reads, what it writes, exit codes, notable flags.

Help text for every command must render in this format:

```
Reads: <input sources>
Writes: <output paths>
Exit: 0=success  1=<error condition>  2=<error condition>
```

### `parallax ingest <path|dir>`

**Reads:** Video files at the given path or directory.  
**Writes:** `ingest/<clip-slug>/transcript.json` and `ingest/<clip-slug>/index.json` per clip. The index records duration, frame rate, resolution, detected language, and a file hash.  
**Exit:** 0=success, 1=file not found or unreadable, 2=API error during transcription.

**Flags:**
- `--visual` — adds per-scene frame analysis via Gemini Vision. Records scene descriptions in the clip's index.
- `--estimate` — dry-run mode. Reads file metadata (duration, count), prints: total duration, estimated processing time, estimated cost per analysis mode (transcription only vs. transcription + visual). Makes zero API calls. Exits 0 always.
- `--json` — emits NDJSON progress events: `{event: "file_start", file: "..."}`, `{event: "file_done", file: "...", duration_sec: N, cost_usd: N}`, `{event: "summary", files: N, total_duration_sec: N, total_cost_usd: N}`.

**Behavior:** When given a directory, processes all video files in parallel (one worker per file, bounded by CPU count). Non-video files are skipped with a warning. Already-ingested clips (hash match in index) are skipped unless `--force` is passed.

**Cost note:** Transcription uses Whisper. Visual analysis uses Gemini. `--estimate` reads from `core/pricing.py` which carries a `LAST_VERIFIED` date — update quarterly.

### `parallax generate still <prompt>`

**Reads:** Optional `--ref <file>` image(s) as visual reference.  
**Writes:** `stills/<slug>.png` per generated image in cwd.  
**Exit:** 0=success, 1=API error, 2=invalid prompt or ref file not found.

**Flags:**
- `--ref <file>` — one or more reference images passed to Gemini.
- `--count N` — number of stills to generate (default: 1, max: 8).
- `--json` — NDJSON: `{event: "generated", file: "stills/foo.png", cost_usd: N}` per image.

**Backend:** Gemini image generation.

---

### `parallax generate video <prompt>`

**Reads:** Optional `--ref <file>` image as motion reference.  
**Writes:** `input/<slug>.mp4` in cwd.  
**Exit:** 0=success, 1=API error, 2=invalid params.

**Flags:**
- `--ref <file>` — reference image for the video generation.
- `--duration N` — clip duration in seconds (default: 5, provider-dependent max).
- `--json` — NDJSON: `{event: "submitted", job_id: "..."}`, `{event: "done", file: "input/foo.mp4", cost_usd: N}`.

**Backend:** fal.ai `kling-video/v3/pro`. Polls for completion. Prints job ID on submission so the user can check status independently.

---

### `parallax generate voice <script>`

**Reads:** Inline script text, or `--from-manifest` reads VO lines from `cwd/manifest.yaml`.  
**Writes:** `audio/<slug>.mp3` per voice segment.  
**Exit:** 0=success, 1=API error, 2=manifest missing when `--from-manifest` is used.

**Flags:**
- `--from-manifest` — reads all `vo` fields from manifest scenes.
- `--voice <id>` — ElevenLabs voice ID override. Falls back to project default from manifest.
- `--json` — NDJSON: `{event: "segment_done", file: "audio/foo.mp3", duration_sec: N, cost_usd: N}` per segment.

**Backend:** ElevenLabs TTS.

---

### `parallax script write <brief>`

**Reads:** Nothing from disk. Brief is the inline argument.  
**Writes:** Nothing by default — prints script to stdout.  
**Exit:** 0=success, 1=API error.

**Flags:**
- `--out <file>` — writes to file instead of stdout.
- `--json` — wraps output in `{event: "script", content: "..."}`.

**Behavior:** Calls the writing agent (Claude). The brief can be a quoted string or piped via stdin. No file side effects unless `--out` is passed — this keeps the command composable.

---

### `parallax script rewrite`

**Reads:** `--from <file>` (required) or stdin.  
**Writes:** Revised script to stdout by default.  
**Exit:** 0=success, 1=API error, 2=source file not found.

**Flags:**
- `--from <file>` — source script file.
- `--out <file>` — write result to file.
- `--instruction <text>` — rewrite directive (e.g., "make it 30 seconds shorter", "shift tone to urgent").

---

### `parallax edit <instruction>`

**Reads:** `cwd/manifest.yaml`.  
**Writes:** Modified `cwd/manifest.yaml` in-place.  
**Exit:** 0=success, 1=manifest not found, 2=editor agent error, 3=validation failure on result.

**Behavior:** Natural language instruction is routed to an editor agent that modifies the manifest YAML. After modification, the manifest is validated (same rules as `manifest validate`). Prints a unified diff of the changes before writing. With `--dry-run`, prints the diff but does not write.

**Flags:**
- `--dry-run` — show diff, do not write.
- `--json` — emits `{event: "diff", patch: "..."}` then `{event: "written"}`.

---

### `parallax compose`

**Reads:** `cwd/manifest.yaml`.  
**Writes:** `output/<project-name>_<timestamp>.mp4`.  
**Exit:** 0=success, 1=manifest missing or invalid, 2=ffmpeg render error.

**Behavior:** Pure ffmpeg execution — zero AI calls. Reads each scene's still, applies Ken Burns motion parameters from the manifest, splices voiceover audio, burns captions at the timecodes specified in the manifest, and concatenates all scenes into the final output file. If any source file referenced in the manifest is missing, exits 1 with a clear error listing every missing file before attempting render.

**Ken Burns parameters per scene:** `zoom_start`, `zoom_end`, `pan_x`, `pan_y`, `duration_sec`. Defaults live in `core/defaults.py` and are documented in the manifest schema.

**Flags:**
- `--preview` — renders at 480p, skips caption burn, faster. Writes to `output/preview_<timestamp>.mp4`.
- `--scene <N>` — renders only scene N (1-indexed). Useful for spot-checking a single scene without running the full timeline.
- `--json` — NDJSON: `{event: "scene_start", scene: N}`, `{event: "scene_done", scene: N, duration_sec: N}`, `{event: "compose_done", file: "output/foo.mp4", total_sec: N}`.

---

### `parallax trim <file>`

**Reads:** The specified video or audio file.  
**Writes:** `<original-name>_trim.<ext>` alongside the original.  
**Exit:** 0=success, 1=file not found, 2=ffmpeg error.

**Flags:**
- `--start <timecode>` — trim start point (HH:MM:SS.ms or seconds).
- `--end <timecode>` — trim end point.
- `--tail <seconds>` — remove N seconds from the end (useful for cutting recording tails).

If no flags are passed, launches an interactive mode: plays the file in the terminal player, prompts for start and end points, confirms before writing.

---

### `parallax manifest <subcommand>`

All subcommands read and/or write `cwd/manifest.yaml`.

**`manifest show`** — pretty-prints the current manifest. Accepts `--scene <N>` to show one scene.

**`manifest set <key> <value>`** — sets a top-level or scene-level key. Example: `manifest set project.voice_id Rachel`. Dot-notation for nested keys.

**`manifest add-scene`** — appends a new empty scene to the manifest's scene list. Accepts `--after <N>` to insert after scene N.

**`manifest remove-scene <N>`** — removes scene N (1-indexed) from the manifest. Prompts for confirmation unless `--yes` is passed.

**`manifest validate`** — validates the manifest against the schema. Prints all validation errors. Exit 0 if valid, exit 1 if invalid.

**Exit codes for all subcommands:** 0=success, 1=manifest not found, 2=validation/schema error.

---

### `parallax status`

**Reads:** `cwd/manifest.yaml`.  
**Writes:** Nothing.  
**Exit:** 0 always (informational command).

**Output:** Prints a summary block:
- Project name and output path
- Scene count
- Stills: N present, N missing (with file paths for missing)
- Voiceover: N segments present, N missing
- Output file: present (with path and timestamp) or absent
- Estimated compose time (based on total scene duration from manifest)

With `--json`, emits a single JSON object with all fields.

---

### `parallax web`

**Reads:** Nothing.  
**Writes:** Nothing (server writes to its own log path).  
**Exit:** 0 on clean shutdown, 1 on port conflict or startup failure.

Launches the parallax-web server. Replaces `parallax chat`. Registers in `~/.parallax/servers.json` on start, deregisters on SIGINT/SIGTERM/normal exit. Prints the local URL on startup.

**Flags:**
- `--port <N>` — override default port.
- `--open` — open the URL in the default browser after startup.

---

### `parallax project new <name>`

**Reads:** Nothing.  
**Writes:** `~/Documents/parallax-projects/<name>/` with subdirectories: `input/`, `output/`, `drafts/`, `stills/`, `audio/`, `logs/`, and a starter `manifest.yaml`.  
**Exit:** 0=success, 1=project already exists, 2=write error.

The starter manifest includes the project name, creation date, an empty scenes list, and commented documentation of every available manifest field.

---

### `parallax project list`

Lists all directories under `~/Documents/parallax-projects/`. For each project, shows: name, creation date, last modified date, whether a manifest is present, whether output files exist.

---

### `parallax setup`

Interactive setup wizard. Checks:
1. Required system deps: `ffmpeg`, `python 3.11+`, `just`. Prints version for each found, error for each missing.
2. API key configuration (BYOK mode): checks for `ANTHROPIC_API_KEY`, `AI_VIDEO_GEMINI_KEY`, `ELEVENLABS_API_KEY`, `FAL_KEY` in environment. Reports present/missing for each.
3. Parallax account login (managed billing mode): alternative to BYOK. If logged in, shows account status.
4. Default ingest mode preference: prompts user to set their default (`transcription-only` vs. `transcription+visual`). Writes to `~/.parallax/config.toml`.

Non-interactive: `--check` flag runs all checks and exits 0 if everything is ready, 1 if anything is missing. No prompts.

---

### `parallax update`

Checks for a newer version of the Parallax CLI. If found, downloads and installs it. Prints current version and latest version. Exits 0 if already up to date.

---

### `parallax publish`

Stub for V3. Prints: "Publishing to external platforms is coming in Parallax V3." Prints a buymeacoffee link. Exits 0.

## Manifest Schema

The manifest is the central data structure of a Parallax project. It is a YAML file that describes the complete production: scenes, assets, voiceover, captions, and render parameters.

### Top-Level Fields

```yaml
project:
  name: string          # required
  voice_id: string      # ElevenLabs voice ID
  output_resolution: string  # e.g. "1920x1080", "1080x1920", "1080x1080"
  output_fps: integer   # default: 24
  caption_style: string # "none" | "bottom" | "dynamic"

scenes:
  - scene: integer      # scene number, 1-indexed
    still: string       # path to still image (relative to project root)
    audio: string       # path to VO audio (relative to project root)
    vo: string          # voiceover script text (source for generate voice)
    duration_sec: float # scene duration in seconds
    caption: string     # caption text (if different from vo)
    ken_burns:
      zoom_start: float # e.g. 1.0
      zoom_end: float   # e.g. 1.15
      pan_x: float      # -1.0 to 1.0 (left to right)
      pan_y: float      # -1.0 to 1.0 (top to bottom)
```

Required fields: `project.name`, `scenes[*].scene`, `scenes[*].duration_sec`. All others are optional but generate warnings in `manifest validate` if missing when they are needed for the current capability mode.

### Manifest Validation Rules

- Scene numbers must be sequential starting from 1.
- `duration_sec` must be > 0.
- `output_resolution` must match `WxH` pattern.
- `output_fps` must be in `[12, 24, 30, 60]`.
- `caption_style` must be one of the allowed values.
- `ken_burns` zoom values must be in `[0.8, 2.0]`.
- If `audio` is set, the file must exist at compose time (warn at validate time).
- If `still` is set, the file must exist at compose time (warn at validate time).

## V2 Capability Modes

The web agent (accessible via `parallax web`) routes user requests into one of six capability modes. Each mode has a defined command sequence and cost profile.

### Mode 1: Stills Only

Goal: generate images, no video output.

Command sequence: `generate still`

Cost: Gemini image generation only. No ElevenLabs, no fal.ai, no ffmpeg.

When to suggest: user asks for images, thumbnails, brand assets, or says they don't need video.

---

### Mode 2: Ken Burns Draft

Goal: stills + motion → draft video, no live footage.

Command sequence: `generate still` → `manifest set` / `edit` → `compose`

Cost: Gemini image generation + ffmpeg (local, no API cost).

When to suggest: user wants a polished draft from scratch with no footage, or wants to prototype a concept cheaply before committing to voice generation.

---

### Mode 3: Storyboard

Goal: produce a timed scene plan and VO script — no generation calls beyond the writing agent.

Command sequence: `script write` → `manifest` (populate scenes + VO text manually) → `status`

Cost: LLM call only (script write). Zero image, audio, or video API spend.

When to suggest: user is early in ideation, wants to see a structured plan before spending money, or is working with a client and needs something to review.

---

### Mode 4: Full Production

Goal: end-to-end ad from raw brief or footage.

Command sequence: `ingest` → `generate still` → `generate voice` → `edit` → `compose`

Cost: Whisper (transcription) + Gemini (stills) + ElevenLabs (voice) + ffmpeg (local).

When to suggest: user has clear brief, possibly has footage for reference, wants a complete deliverable.

---

### Mode 5: Footage Edit

Goal: edit existing footage, no generation.

Command sequence: `ingest` → `edit` → `compose`

Cost: Whisper (transcription, optional) + ffmpeg (local). No image or voice generation cost.

When to suggest: user has existing footage and wants it cut, trimmed, captioned, or restructured.

---

### Mode 6: Script First

Goal: start from copy, build outward.

Command sequence: `script write` → `generate voice` → `generate still` → `compose`

Cost: LLM (script) + ElevenLabs (voice) + Gemini (stills) + ffmpeg.

When to suggest: user starts by describing what they want to say, not what they want to show.

## TEST_MODE Specification

V2 must support a `TEST_MODE=true` environment variable that enables fully stubbed execution. The goal is end-to-end pipeline verification without spending money on generative APIs.

### Rules

**Scope:** TEST_MODE affects all generative API calls: Gemini image generation, Gemini vision/frame analysis, ElevenLabs TTS, fal.ai video generation, and Whisper transcription. It does not stub ffmpeg — compose and trim run real ffmpeg against stub inputs.

**Stub outputs must be real files.** Stubs write actual files to disk so downstream commands have something to work with:
- Image stubs: solid-color PNG at the project's output resolution (default: 1920x1080, configurable).
- Audio stubs: silent WAV of the expected duration (calculated from VO text character count at 150 WPM).
- Video stubs: a short solid-color MP4 (3 seconds, 24fps) generated by ffmpeg locally.
- Transcript stubs: a deterministic `transcript.json` with placeholder text derived from the input filename.

**Determinism:** Given the same inputs (filename, prompt text, duration), TEST_MODE always produces the same stub output. This ensures reproducible test runs.

**Cost/time estimates:** `ingest --estimate` in TEST_MODE uses real duration calculation from file metadata but reports latency as 0ms for API calls. Reported costs use the same `core/pricing.py` rates as real mode — estimates in TEST_MODE are accurate cost projections, not zeroed out.

**Output directory:** TEST_MODE writes to a temp directory (system temp + `parallax-test-<timestamp>/`). The temp dir is reported at the start of each run and cleaned up after unless `--keep-test-output` is passed.

**Structured events:** All `--json` NDJSON events still fire in TEST_MODE, with `"test_mode": true` added to each event. This allows test harnesses to verify event shape without real API calls.

### Test Coverage Requirement

Every `just test` run must exercise at least one complete end-to-end flow per capability mode:
- Mode 1: `generate still` with stub → verify PNG written to `stills/`.
- Mode 2: `generate still` → `manifest add-scene` → `compose` → verify MP4 written to `output/`.
- Mode 3: `script write` → `manifest` population → `status` → verify status output is valid.
- Mode 4: `ingest` → `generate still` → `generate voice` → `edit` → `compose` → verify final MP4.
- Mode 5: `ingest` → `edit` → `compose` → verify final MP4.
- Mode 6: `script write` → `generate voice` → `generate still` → `compose` → verify final MP4.

Tests must also cover:
- `manifest validate` with a valid manifest (exit 0).
- `manifest validate` with an intentionally broken manifest (exit 1, error message present).
- `ingest --estimate` exits 0, makes zero API calls, prints cost/duration summary.
- `compose` with a missing still file (exit 1 before ffmpeg runs, error lists missing file).
- `--json` flag on `compose` emits valid NDJSON (parseable line by line).

## Agentic Use Contract

Every V2 command is designed for use by the parallax-web agent and external pipeline automation.

**Universal `--json` flag:** All commands support `--json` to emit NDJSON (one JSON object per line) instead of human-readable text. Agents consume NDJSON; humans read the default output.

**Predictable exit codes:** All exit codes are documented in help text and this spec. Exit code semantics do not change between versions without a major version bump.

**Composability:** Commands are designed to be chained via shell pipes. `script write` prints to stdout so its output can be piped to `generate voice` or redirected to a file. `status --json` can be parsed by a monitoring agent.

**Help text contract format:**
```
Reads: <input sources>
Writes: <output paths>
Exit: 0=success  1=<condition>  2=<condition>
```

This format appears in every command's `--help` output. It is the machine-readable API contract for agents that need to understand what a command does before calling it.

**`parallax web` agent routing:** The HoP (Head of Production) agent running in the web interface must be able to:
- Parse user intent and route to the correct capability mode.
- Confirm before any step that makes a paid API call.
- Report estimated cost before confirming.
- Handle partial completions — if a pipeline fails mid-way, the agent reports exactly which step failed and what files were written before the failure.
- Use `--json` on all CLI calls it makes, and parse the NDJSON stream to track progress.

## Onboarding UX

The HoP agent must handle the following first-contact scenarios correctly.

### "What can you do?" / "Help"

Response covers: the six capability modes in plain English, the cost model (which modes spend money on what), and a prompt to share what the user is trying to make. No jargon. No command names in the first response unless the user is clearly technical.

### "I want to make an ad"

Clarify with the minimum necessary questions. Required to know before routing: (1) does the user have footage or are we starting from scratch, (2) does the user have a script or copy in mind or should we write it. That is all — do not ask about format, duration, or tone up front. Those come later when filling out the manifest.

Decision tree:
- Has footage + has script → Mode 5 (footage edit) or Mode 4 depending on whether they want generated stills to supplement.
- Has footage + no script → Mode 5, but open with `script write` to draft VO.
- No footage + has script → Mode 6 (script first).
- No footage + no script → Mode 4 or Mode 3 depending on budget readiness.
- Early ideation, unclear budget → Mode 3 (storyboard) as the zero-cost starting point.

### "I have footage" (first message)

Immediately suggest `parallax ingest`. Explain what ingest does (transcribes, indexes, gives the agent something to work with) before asking what the user wants to do with the footage. Do not ask for the footage path in the web UI — the user uploads it via the file interface.

### Confirmation before API spend

Before any command that makes a paid API call (generate still, generate voice, generate video, ingest with --visual), the agent must:
1. State what it is about to call.
2. State the estimated cost.
3. Ask for explicit confirmation.

This is a hard requirement. Agents must not auto-proceed on generative API calls.

### Storyboard as default for "I'm not sure"

When a user signals uncertainty about their brief, budget, or direction, the agent defaults to Mode 3 (storyboard). This surfaces a structured plan for zero generation cost. The agent explains: "I can map out a complete scene plan and script right now at no cost — we can review it together and only generate images and audio once you're happy with the direction."

## File System Conventions

All path references are relative to the project root unless otherwise noted.

```
<project-root>/
  manifest.yaml         # the manifest
  input/                # raw footage and generate video outputs
  output/               # final composed videos
  stills/               # generate still outputs
  audio/                # generate voice outputs
  drafts/               # intermediate or draft files
  ingest/               # ingest outputs
    <clip-slug>/
      transcript.json
      index.json
  logs/                 # CLI and agent logs
```

Global state (cross-project) lives at:
```
~/.parallax/
  config.toml           # user preferences (default ingest mode, etc.)
  servers.json          # running parallax-web registry
  update_cache.json     # last update check result
```

Projects live at `~/Documents/parallax-projects/<name>/` when created via `project new`. A project root can also be any arbitrary directory — `manifest.yaml` presence is the signal that a directory is a Parallax project.

## Pricing and Cost Tracking

All provider costs are tracked in `core/pricing.py`. The file includes a `LAST_VERIFIED` date at the top. Update quarterly or when a provider changes pricing.

**Instrumentation:** Every external API call emits a `request_intended` event and a `cost_estimated` event before dispatch, using `core/instrumented.py`. Both real and TEST_MODE runs emit identical event shapes. This enables the web interface to show projected spend alongside actual spend.

**NDJSON cost fields:** Commands that make API calls include `cost_usd` on their completion events. The `compose` command includes `cost_usd: 0` since it is pure local ffmpeg.

## V1 → V2 Migration Notes

These notes are for developers migrating V1 internals to V2. They are not user-facing.

- `parallax run` is removed. Its logic is distributed across `ingest`, `generate`, and `compose`.
- `parallax animate` is removed. Ken Burns motion is now handled by `compose` reading `ken_burns` parameters from the manifest.
- `parallax chat` becomes `parallax web`.
- `parallax create` becomes `parallax generate still`.
- `parallax voiceover` becomes `parallax generate voice`.
- `parallax transcribe` is folded into `parallax ingest`.
- `parallax align` is retired. If audio/video alignment is needed in a future version it should live as a flag on `compose`, not a standalone command.
- `parallax projects` (plural) becomes `parallax project` (singular) with `new` and `list` subcommands.
- The `.parallax/` hidden directory at cwd is replaced by the visible `parallax/users/<user>/<project>/` nested workspace (see DEV_LOG 2026-04-14 DECISION entry for rationale). Global state in `~/.parallax/` is unchanged.

## Open Questions (Deferred to V3 or Later)

The following were considered for V2 and explicitly deferred:

- **`parallax publish`** — platform integrations (YouTube, Meta Ads, etc.) are V3. The stub exits cleanly and signals intent.
- **`parallax align`** — audio/video alignment was in V1. Deferred because the use case is narrow and can be handled by user calling `trim` on both files. Revisit if it surfaces repeatedly in user feedback.
- **Multi-user / team projects** — the current layout supports per-user workspaces at the filesystem level but there is no collaboration or sharing model. V3 concern.
- **`ingest --visual` on very long files** — frame analysis at scale is expensive. A `--sample-rate` flag (e.g., analyze one frame per N seconds) would control cost. Deferred; add to pricing documentation as a caveat for now.
- **Preset system** — a `parallax preset` command for saving and reusing manifest templates across projects. Useful for agencies. Not in V2 scope.
