**Version:** 2.3.0  
**Date:** 2026-04-15  
**Status:** Draft  
**Supersedes:** v0.1.0 (2026-04-11)

## Overview

Parallax is a Plexi app that is primarily a **visualizer and control surface for the Parallax CLI**. The app's job is to make the output of `parallax` tangible — you see your stills, clips, and assembled videos inline, you can inspect and edit the manifest YAML directly, and you can drive the full pipeline agentically via the HoP chat interface.

It is not a general-purpose video editor. It does not replace Premiere or DaVinci. What it replaces is the cycle of: run CLI → find output → open in Finder → play → go back to terminal → re-run. That cycle disappears. Everything is in one surface.

**Primary use cases in priority order:**

1. **Agentic production** — chat a brief to HoP, approve stages, get a video out.
2. **Media visualization** — browse stills, clips, and assembled output generated by the CLI. See what you have.
3. **Manifest editing** — open `.parallax/manifest.yaml` in a clip editor view, drag scenes, adjust timing, change volume, swap stills — and run compose from within the app.
4. **Footage triage** — open a folder of raw footage, skim clips, mark selects, let HoP build a manifest from your selections.

## 1. Layout

The app uses a three-panel layout. All panels are resizable.

```
┌──────────────────────────────────────────────────────────────────────┐
│ TOOLBAR: [Parallax]  [project-dir]              [tier] [$cost] [⚙]  │
├────────────┬─────────────────────────────────────────────────────────┤
│            │                                                          │
│  SIDEBAR   │                  MAIN PANEL                             │
│            │                                                          │
│  Media     │   (switches between: Chat / Clip Editor / Viewer)       │
│  Library   │                                                          │
│            │                                                          │
├────────────┴─────────────────────────────────────────────────────────┤
│ STATUS BAR: [stage] [progress] ............... [session $] [total $] │
└──────────────────────────────────────────────────────────────────────┘
```

**Toolbar** — minimal per Plexi convention. App name left, gear icon right (Plexi-rendered settings). Middle shows current project dir basename + current intelligence tier + session cost.

**Sidebar** — Media Library (see §3). Persistent across main panel views.

**Main Panel** — switches between three views: Chat (§2), Clip Editor (§4), Viewer (§5). Switching is instant; state in each view is preserved.

**Status Bar** — active stage name (e.g. "composing…"), progress indicator, running session cost, running project total.

## 2. Chat / HoP Interface

The primary agentic interface. Users talk to HoP in natural language. HoP never edits YAML directly — it dispatches to the Editor agent, which owns the manifest.

### 2.1 Message Types

| Type | Appearance |
|------|-----------|
| User message | Right-aligned, plain text |
| HoP response | Left-aligned, prose |
| System event | Centered, dimmed — e.g. "✓ 4 stills generated" |
| Stage approval gate | Full-width card with Approve / Reject buttons |
| Error | Red border, full error text visible |

### 2.2 Stage Approval Gates

Every pipeline stage produces an approval gate before advancing. HoP never auto-advances past an approval. Gates are inline in the chat — no modal, no separate screen.

| Stage | What the gate shows |
|-------|-------------------|
| Script | Script text, word count, estimated voiceover duration |
| Storyboard | Shot descriptions, scene count |
| Stills | Thumbnail grid of generated stills (inline, from Media Library) |
| Manifest draft | Read-only YAML preview of the generated manifest |
| Ken Burns preview | Playable preview (silent, motion only) in the Viewer panel |
| Voiceover | Audio waveform + playback for each segment |
| Captions | Captioned video preview |
| Final assembly | Playable final video, file size, cost total for this run |

Rejecting a gate returns control to HoP with a brief text box pre-focused for feedback. HoP revises and re-submits the stage.

### 2.3 Commands

| Command | Action |
|---------|--------|
| `/brief` | Force-route the next message as a job brief to HoP |
| `/edit` | Open `.parallax/manifest.yaml` in Clip Editor view |
| `/compose` | Run `parallax compose` from current manifest |
| `/tier low\|medium\|high` | Set intelligence tier |
| `/open [path]` | Open a video/still in the Viewer |
| `/footage [dir]` | Load a footage folder into the Media Library for triage |
| `/cost` | Show cost breakdown for the session |

### 2.4 Slash-Agent Mode

Pressing `/` at the start of an empty prompt enters agent mode (standard Plexi behavior). In agent mode, Parallax exposes these tools to the agent:

- `list_media` — list stills, clips, output files in the project
- `read_manifest` — return current manifest.yaml as text
- `write_manifest` — write a new manifest.yaml (Editor agent only)
- `run_compose` — trigger `parallax compose`
- `run_stage` — trigger a specific pipeline stage (stills, voiceover, captions, etc.)
- `play_clip` — load a file into the Viewer
- `get_cost` — return current cost totals

## 3. Media Library (Sidebar)

The sidebar is a persistent media browser scoped to the current project directory. It watches the filesystem and updates in real time as the CLI generates new files.

### 3.1 Sections

```
▼ Output
    final_v1.mp4        [▶]
    final_v2.mp4        [▶]
▼ Stills
    scene_01.png        [▶]
    scene_02.png        [▶]
    scene_03.png        [▶]
▼ Clips
    raw_clip_01.mp4     [▶]
    raw_clip_02.mp4     [▶]
▼ Audio
    vo_segment_01.mp3   [▶]
    vo_segment_02.mp3   [▶]
▼ Footage  (only shown if /footage was used)
    IMG_4231.MOV        [▶]  [★]
    IMG_4232.MOV        [▶]  [★]
```

- Clicking any item loads it into the Viewer panel.
- `[★]` marks a clip as a select (for footage triage — see §6).
- Double-clicking a still in the Stills section opens it in the Clip Editor with that scene focused.
- The Manifest section shows the current `.parallax/manifest.yaml` as a clickable item — opens Clip Editor.

### 3.2 Real-Time Watching

The sidebar uses filesystem watching (via `watchdog` or equivalent) on:
- `stills/`
- `output/`
- `audio/`
- `input/` (for footage triage)
- `.parallax/manifest.yaml`

New files appear instantly. No manual refresh.

## 4. Clip Editor

The Clip Editor is a manifest-first visual editor. It represents the state of `.parallax/manifest.yaml` as a visual timeline/scene list. Editing in the Clip Editor modifies the manifest YAML directly. There is no intermediate in-memory representation — the YAML is the single source of truth.

### 4.1 Scene List View

The primary Clip Editor view is a vertical scene list (not a horizontal timeline — horizontal timelines are hard to use for long-form content and don't add value here).

Each scene row shows:

```
┌─────────────────────────────────────────────────────────────────────┐
│ [thumbnail]  scene_01.png                              [duration: 4s]│
│              motion: zoom_in_slow   ·  transition: cut              │
│              caption: "The moment everything changed."              │
│              [vol: ██████░░░░ 65%]                                  │
└─────────────────────────────────────────────────────────────────────┘
```

Controls per scene row:
- **Thumbnail** — click to preview in Viewer. Drag thumbnail to reorder scenes.
- **Still path** — click to open a file picker (scoped to `stills/`).
- **Duration** — editable inline (number field).
- **Motion** — dropdown: `zoom_in_slow`, `zoom_out_slow`, `pan_left`, `pan_right`, `static`, `zoom_in_fast`.
- **Transition** — dropdown: `cut`, `fade`, `dissolve`.
- **Caption** — editable inline text field.
- **Volume** — horizontal slider, 0–100%. Applies to any audio segment attached to this scene.
- **Delete** — `×` on hover.

### 4.2 Global Settings Panel

Collapsed by default, expandable at the top of the Clip Editor:

- Output resolution (dropdown: 1080p, 4K, 9:16, 1:1)
- Background music path + volume slider
- Global voiceover volume slider
- Caption style (font, size, position — mapped to manifest `caption_style` block)
- Headline text + position

### 4.3 Raw YAML Toggle

A `{ }` button in the toolbar switches the Clip Editor between the visual scene list and a raw YAML text editor (monospace, line-numbered, syntax highlighted). Both views write to the same file. Switching between views re-parses the YAML — if the YAML is invalid, the visual view shows an error banner and blocks switching back until fixed.

This is the escape hatch for power users who want to hand-edit the manifest, add fields the visual editor doesn't expose, or paste a manifest from another project.

### 4.4 Compose Button

A persistent `▶ Compose` button at the bottom of the Clip Editor runs `parallax compose` from the current manifest. Output appears in the Viewer when done. The button shows a spinner and disables during an active compose run. Errors are shown inline below the button.

### 4.5 Dirty State

If the manifest has unsaved changes, the Clip Editor header shows a `●` dot. Changes auto-save to disk on every edit (no Save button needed — the YAML is the live file). The dirty indicator tracks whether the current manifest has been composed yet: it clears after a successful compose run, re-appears on any edit.

## 5. Viewer Panel

The Viewer replaces the "open in Finder / preview in QuickTime" loop. It's an inline media player.

### 5.1 Supported Media

| Type | Display |
|------|---------|
| MP4 / MOV | Video player with playback controls |
| PNG / JPG | Still image, zoom on hover |
| MP3 / WAV | Waveform + playback controls |

### 5.2 Playback Controls

Standard transport controls: play/pause (space), scrub bar, current time / total duration, volume. No frame-by-frame needed — this is a preview player, not an editor.

### 5.3 Side-by-Side Compare

If two videos are loaded (e.g. `final_v1.mp4` and `final_v2.mp4`), the Viewer can show them side by side with synchronized playback. Triggered by shift-clicking a second item in the Media Library.

### 5.4 Auto-Open After Compose

After every successful `parallax compose` run, the output video is automatically loaded into the Viewer (matching the existing "always open video after rendering" rule). This is non-optional behavior.

## 6. Footage Triage

For editing projects that start from raw footage rather than generated stills. Activated via `/footage <dir>` in chat or by dragging a folder onto the app.

### 6.1 Flow

1. User runs `/footage ~/Desktop/wedding-footage/`
2. Media Library populates the Footage section with all `.mp4 / .mov` files in that directory.
3. Clicking a clip loads it in the Viewer with scrub bar.
4. User marks selects with `★` (keyboard shortcut: `S`). Can add a short text note per clip.
5. When the user is done selecting, they type "build a manifest from my selects" in chat.
6. HoP reads the select list + notes, generates a brief, dispatches the Editor agent to produce a manifest using the selected clips as `type: raw_clip` scenes.
7. Clip Editor opens with the generated manifest.

### 6.2 iPhone MOV Handling

Raw iPhone footage is automatically detected (1920×1080 with Display Matrix rotation tag) and flagged in the Media Library with a rotation badge. The Viewer plays them correctly rotated. When the Editor agent generates a manifest scene for an iPhone clip, it automatically sets `transpose: 2` and `size: 1080x1920` per the established gotcha.

## 7. Pipeline Stage Panels

When a pipeline is running (HoP-dispatched or manual compose), the Status Bar expands to show a stage progress bar:

```
[script ✓] → [stills ✓] → [manifest ✓] → [compose ⟳] → [captions] → [assembly]
```

Clicking a completed stage jumps the Main Panel to the relevant view (e.g. clicking "stills ✓" opens the still grid in the Viewer). Each stage that produced approvable output shows the approval gate inline in Chat.

## 8. agentic AI Video Generation (fal.ai)

When the project needs AI-generated video clips (not just Ken Burns on stills), HoP can dispatch `parallax veo` to generate real video via fal.ai.

**Trigger:** User says something like "generate a clip of a woman walking through a market" or includes `type: ai_video` scenes in a manifest.

**Flow:**

1. HoP confirms: scene prompt, duration (3–15s), aspect ratio, estimated cost.
2. User approves. A `type: ai_video` scene with `prompt:` and `duration:` is written to the manifest by the Editor agent.
3. `parallax compose` sees `type: ai_video` scenes and calls `fal_client.subscribe("fal-ai/kling-video/v3/pro/image-to-video", ...)`.
4. Generated clips are saved to `clips/` and the manifest `src:` field is updated to point to the downloaded file.
5. Clip appears in Media Library → Viewer auto-opens.

**Cost note:** ~$0.34/sec. HoP always shows the estimated cost before dispatching.

## 9. State Management

Follows the four-bucket Plexi state model:

| Bucket | Contents |
|--------|---------|
| `user_state` | Conversation history, manifest edits (undo-able, sync-able) |
| `derived` | Viewer playhead position, active stage, compose progress |
| `session` | Footage triage selects, current open view (Chat/Editor/Viewer) |
| `persistent` | Project cost totals, intelligence tier preference, last-opened project dir |

## 11. What This Is Not

Explicit non-goals — features not in scope and why:

- **No timeline editor.** Horizontal drag-to-trim timelines are complex to build and add little value when the LLM can make timing decisions. The Clip Editor is a manifest editor, not a DAW.
- **No color grading / filters.** Handled in ffmpeg filters in the manifest if needed. The visual editor doesn't need to show a color wheel.
- **No multi-track audio mixing.** Volume sliders per scene + global background music volume covers 95% of needs. Full mixing is Premiere's job.
- **No cloud sync.** The project directory is the source of truth. Sync is a filesystem problem (Dropbox, git, etc.) not an app problem.
- **No export presets UI.** Output format is specified in the manifest `output:` block. The raw YAML toggle is the escape hatch.
- **No frame-accurate trimming UI.** Raw footage trimming is done via manifest `trim_start` / `trim_end` fields (seconds). Precision sub-frame editing is not a use case.

## 12. Protocol Dependencies

The Clip Editor (drag-to-reorder, volume sliders) and Viewer (inline video/audio playback, waveform display) depend on four primitives not yet in the Plexi SDK. These are fully designed and deferred to **Plexi v2.2**, timed with the video editor / DAW work:

- `ctx.slider` — volume sliders per scene, scrub bar
- `ctx.sortable_list` — drag-to-reorder scene list
- `ctx.media_player` — inline video/audio playback (requires AVFoundation in host)
- `ctx.waveform` — audio peak rendering with playback cursor

See `PLEXI/docs/specs/proposals/media-primitives.md` for the full bridge spec.

**Until v2.2 ships:** the Clip Editor renders scene rows with text-only volume display (no slider), reordering is done via the raw YAML toggle, and the Viewer shows `ctx.video_thumbnail` (click-to-open-in-system-player) instead of inline playback.
- **Footage triage selects persistence** — should selects persist to `.parallax/selects.json` or stay session-only? Session-only is simpler; persist if the user ever has a multi-day triage session.
- **fal.ai cost cap** — AI video generation is expensive. Should there be a separate per-session cap for `type: ai_video` scenes, distinct from the general `max_session_usd`?
